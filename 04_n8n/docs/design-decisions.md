# 설계 결정서 (Design Decisions)

> PB 뉴스 요약 자동화 워크플로우의 핵심 설계 선택과 그 이유를 정리한 문서입니다.
> 모든 산출물(n8n 워크플로우, Notion DB, README)은 이 문서의 결정을 따릅니다.

---

## 1. 도구 스택

| 구분     | 선택                                  | 이유                                                                               |
| ------ | ----------------------------------- | -------------------------------------------------------------------------------- |
| 자동화 툴  | **n8n**                             | 워크플로우를 JSON으로 내보내/가져올 수 있어 버전 관리·재현이 쉽고, 셀프호스팅 시 실행 비용이 무료. 노드 기반이라 무코드 요건도 충족.  |
| 생성형 AI | **OpenAI `gpt-4o-mini`**            | 과제 제약(OpenAI API 또는 준하는 모델) 충족. 요약 작업에는 충분한 품질이면서 토큰 단가가 낮아 1일 1회 호출 비용이 무시할 수준. |
| 저장소    | **Notion 데이터베이스**                   | 과제 권장 도구. 속성(Property) 기반이라 제목/요약/링크/날짜를 구조적으로 매핑하기 적합.                          |
| 알림(에러) | **n8n 내장 로그 + (선택) Slack/Email 노드** | 실패 시 사람이 인지할 수 있는 최소 채널 확보.                                                      |

## 2. 데이터 수집 (RSS)

### 2.1 RSS 소스

- **주 피드(Primary):** TechCrunch — `https://techcrunch.com/feed/`
- **대체 피드(Fallback/예비):** Google News RSS 검색 — `https://news.google.com/rss/search?q=AI+when:1d&hl=ko&gl=KR&ceid=KR:ko`

**선택 이유**
- TechCrunch는 RSS를 공식 제공하며 `content:encoded`에 본문 상당량이 포함되어 **요약 입력 품질**이 좋습니다. (과제 금지사항인 "RSS 없는 사이트 강제 크롤링"을 피함)
- 일반 기술 피드를 받아 **키워드 필터로 AI 주제만 추리는 구조**라, "주제 필터링" 단계의 동작을 명확히 보여줄 수 있습니다.
- Google News RSS는 검색 기반이라 AI 항목 확보가 안정적이어서 예비 소스로 둡니다.

### 2.2 주제 필터링 기준

- **필터 키워드(대소문자 무시, 제목+요약 대상):**
  `AI`, `artificial intelligence`, `machine learning`, `deep learning`, `LLM`, `GPT`, `OpenAI`, `Anthropic`, `Gemini`, `neural`, `generative`, `인공지능`, `생성형`, `머신러닝`, `딥러닝`
- **선택 로직:** 키워드를 하나라도 포함하는 항목들 중 **발행일시가 가장 최신인 1건**을 선택.

**선택 이유**
- AI 도메인의 핵심 용어를 한/영 혼용으로 폭넓게 잡아 누락을 줄입니다.
- "1건만 선택"은 과제 요구(1건 추출)와 비용 절감(요약 1회) 원칙을 동시에 만족합니다.
- 최신순 선택은 "오늘의 대표 뉴스"라는 사용자 기대에 부합합니다.

## 3. AI 가공 (요약)

| 항목          | 값                      | 이유               |
| ----------- | ---------------------- | ---------------- |
| 모델          | `gpt-4o-mini`          | 저비용·충분한 요약 품질    |
| 호출 횟수       | **기사 1건당 1회** (재시도 제외) | 과제 비용 원칙 준수      |
| 출력 형식       | **한국어 3줄 이내 불릿 요약**    | 과제 요구(3줄 이내)     |
| temperature | `0.3`                  | 요약의 일관성·사실성 우선   |
| max_tokens  | `300`                  | 3줄 요약에 충분, 비용 상한 |

**요약 프롬프트(요지)**
```
당신은 기술 뉴스 요약가입니다. 아래 기사를 한국어로 3줄 이내,
각 줄을 '- '로 시작하는 불릿으로 요약하세요. 과장·추측 없이 사실만 담으세요.
```

## 4. 자동 저장 (Notion DB 스키마)

| 속성명     | 타입               | 매핑 대상      | 비고               |
| ------- | ---------------- | ---------- | ---------------- |
| `제목`    | Title            | 기사 제목      | 필수               |
| `요약`    | Rich text        | AI 3줄 요약   | 필수               |
| `원문 링크` | URL              | 기사 원문 URL  | 필수 / **중복 방지 키** |
| `발행일시`  | Date             | 기사 발행 시각   | 필수               |
| `출처`    | Rich text        | 피드/매체명     | 추적용              |
| `주제 태그` | Multi-select     | 매칭된 키워드 태그 | 필터 근거 기록         |
| `감성`    | Select(긍정/중립/부정) | 감성 분석 결과   | **보너스 2**        |
| `썸네일`   | Files & media    | 생성 이미지     | **보너스 1**        |
| `상태`    | Select(저장됨/에러)   | 처리 상태      | 운영 모니터링          |
| `수집 시각` | Created time     | 자동         | 감사 로그            |

**선택 이유**
- 과제 필수 4항목(제목·요약문·원문 링크·발행일시)을 각각 **독립 속성**으로 두어 요구를 정확히 충족.
- `원문 링크`를 중복 방지 키로 재사용하여 별도 컬럼 없이 멱등성 확보(아래 5장).
- 보너스(감성·썸네일)는 속성만 미리 만들어 두어, 워크플로우 확장 시 스키마 변경 없이 채울 수 있게 함.

## 5. 중복 방지 / 비용 정책

- **중복 키:** `원문 링크`(URL). 저장 직전 Notion DB를 `원문 링크 == 현재 링크` 조건으로 조회하여, **이미 존재하면 저장·요약을 건너뜀**.
  - URL은 추적 파라미터(`?utm_...`) 제거 후 정규화하여 동일 기사 변형을 흡수.
  - 대안으로 RSS `guid`도 사용 가능하나, 매체 간 일관성이 높은 링크를 1차 키로 채택.
- **요약 호출:** 중복 체크를 **요약보다 먼저** 수행 → 이미 저장된 기사는 OpenAI를 호출하지 않음(비용 0).
- **재시도 상한:** 네트워크성 노드(RSS·OpenAI·Notion)에 한해 **최대 2회 재시도**(총 3회 시도), 시도 간 5초 대기.

## 6. 스케줄 / 안정성 / 에러 처리

### 6.1 스케줄 트리거
- **매일 09:00 (Asia/Seoul)**, cron `0 9 * * *`.
- 자가 재트리거 경로 없음 → **무한 루프 원천 차단**(과제 금지사항).

### 6.2 에러 처리 정책 (3단계)

| 상황                         | 처리                                                     | 이유                                          |
| -------------------------- | ------------------------------------------------------ | ------------------------------------------- |
| **수집 0건 / 필터 결과 0건**       | 에러가 아닌 **정상 분기**로 처리, "오늘 해당 뉴스 없음" 알림 후 종료            | 뉴스 부재는 실패가 아님. 불필요한 에러·재시도 방지               |
| **일시적 실패**(타임아웃·5xx·레이트리밋) | **자동 재시도 최대 2회**(5초 간격)                                | 일시 장애는 재시도로 대부분 복구. 상한으로 비용 폭주 차단           |
| **재시도 후에도 실패**             | 해당 건 **로그 기록 + 스킵**, 운영자 **알림 발송**, 워크플로우는 비정상 종료하지 않음 | 1건 실패가 전체 파이프라인을 멈추지 않게 함(격리). 사람은 사후 인지 가능 |

### 6.3 멱등성
- 동일 시각 중복 실행되어도 5장의 중복 키로 인해 **중복 저장이 발생하지 않음**.

## 7. 보안

- OpenAI 키, Notion 토큰은 **n8n Credentials**(암호화 저장)로 관리하며, 워크플로우 JSON에는 **자격증명 참조 ID만** 포함(평문 키 미포함).
- 로컬 보조 변수는 `.env`로 분리하고 `.gitignore`로 추적 제외. 저장소에는 `.env.example`(마스킹본)만 커밋.
- 문서·스크린샷에는 키/토큰을 `sk-****`, `secret_****` 형태로 **마스킹**.

## 8. 보너스 과제 처리 방침 (선택 구현)

- **보너스 1 (썸네일):** OpenAI 이미지 생성(`gpt-image-1`)으로 기사 주제 기반 썸네일 1장 생성 → `썸네일` 속성에 외부 URL로 첨부.
- **보너스 2 (감성):** 요약 단계에서 동일 응답에 `sentiment: 긍정|중립|부정`을 함께 산출(추가 호출 없이) → `감성` 속성에 매핑.

> 보너스는 스키마·노드 자리만 확보해 두고, 기본 파이프라인 검증 후 활성화하는 것을 권장.
